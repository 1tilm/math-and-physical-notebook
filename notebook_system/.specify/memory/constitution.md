# 学术笔记系统宪法
<!-- Academic Note System Constitution -->

## 核心原则 (Core Principles)

### I. 准确性优先 (Accuracy First)
**所有学术内容必须经过验证，确保科学严谨性**
- 数学公式必须经过推导验证，不允许错误公式
- 物理概念必须符合科学原理，避免概念混淆
- 例题解答必须逻辑清晰，步骤完整
- 引用资料必须标注来源，确保可追溯性
- 任何不确定的内容必须标注"待验证"并及时核实

### II. 结构化组织 (Structured Organization)
**严格遵循Markdown规范v2.0，确保系统一致性**
- 所有笔记必须包含：思维导图、理论基础、考试宝典、易错点数据库、速查手册、例题体系
- 标题层级严格按照规范：# 文档标题 → ## 主要章节 → ### 章节内大节 → #### 具体知识点
- Mermaid思维导图禁止使用数学公式和特殊符号
- 数学公式使用LaTeX语法：行内用$...$，独立公式用$$...$$
- 视觉元素必须按照Emoji规范使用

### III. 错误驱动学习 (Error-Driven Learning) **[NON-NEGOTIABLE]**
**将每个错题转化为学习资源，形成闭环改进**
- 错题必须按照标准模板记录：基本信息、错误分析、正确解答、防错计划
- 错题整合流程：OCR提取 → AI分析 → 自动整合 → 质量验证
- 易错点数据库必须基于实际错题建立，不允许虚构
- 错题复盘必须定期进行：周统计、月分析、学期总结

### IV. AI辅助增强 (AI-Assisted Enhancement)
**利用AI提升效率，但保持人工审核和最终决策权**
- AI生成的内容必须经过人工审核才能发布
- 重要概念和公式推导必须有人工验证
- AI建议作为参考，最终决策权在用户
- 保持学习的主动性，AI仅作为工具而非替代

### V. 渐进式完善 (Progressive Refinement)
**支持增量更新，从基础到高级逐步完善**
- 笔记创建遵循：基础概念 → 核心理论 → 实际应用 → 高级扩展
- 版本控制严格执行：feature分支 → 质量验证 → PR审查 → 合并发布
- 内容更新必须向后兼容，重大变更需要迁移计划
- 鼓励持续改进，但避免频繁的结构性变更

## 质量标准 (Quality Standards)

### 内容质量要求
- **概念解释**：必须准确无误，通俗易懂，避免循环定义
- **公式推导**：必须逻辑清晰，步骤完整，关键步骤有说明
- **例题解答**：必须包含思路分析、详细步骤、结果验证
- **易错点分析**：必须基于实际错题，深入到根本原因
- **实际应用**：必须贴近现实，有具体案例支撑

### 格式质量要求
- **Markdown语法**：严格符合规范，通过自动化验证
- **数学公式**：LaTeX语法正确，渲染效果良好
- **思维导图**：Mermaid语法正确，结构清晰，无数学符号
- **表格格式**：对齐统一，内容简洁，便于阅读
- **视觉效果**：Emoji使用规范，重点突出，层次分明

### 可用性要求
- **查找效率**：目录结构清晰，关键词易于搜索
- **学习效果**：内容组织符合认知规律，难度递进合理
- **复习便利**：速查手册完整，考试宝典实用
- **错题整合**：错题处理及时，分析深入，改进有效

## 开发工作流程 (Development Workflow)

### 笔记创建流程
1. **需求分析**：明确学习目标和内容范围
2. **规格制定**：使用`/speckit.plan`创建详细规格说明
3. **任务分解**：使用`/speckit.tasks`分解具体任务
4. **内容实施**：使用`/speckit.implement`自动化创建
5. **质量验证**：运行`validate-note.sh`进行全面检查
6. **审查发布**：通过PR流程进行代码审查

### 错题处理流程
1. **错题收集**：拍照或截图保存错题
2. **模板填写**：按照标准模板记录错题信息
3. **AI分析**：使用AI进行错误分析和解答生成
4. **人工审核**：验证AI分析的准确性和完整性
5. **自动整合**：运行`integrate-errors.sh`整合到笔记
6. **效果跟踪**：更新错题统计和学习建议

### 版本控制规范
- **分支命名**：`feature/学科-主题`，如`feature/数学-线性代数`
- **提交信息**：`feat: 添加XXX详解笔记` 或 `fix: 修正XXX公式错误`
- **PR标题**：`新增：XXX详解笔记` 或 `修正：XXX内容错误`
- **审查要求**：所有PR必须通过自动化验证和人工审查
- **发布标准**：通过所有质量检查，获得审查批准

## 技术约束 (Technical Constraints)

### 文件组织要求
- **目录结构**：严格按照Spec-Kit标准组织
- **命名规范**：使用中文命名，统一"详解.md"后缀
- **文件大小**：单个笔记文件不超过5MB，超出需要拆分
- **图片资源**：统一存放在`图片资源/`目录，使用相对路径引用

### 自动化工具要求
- **脚本执行**：所有脚本必须在Linux/macOS/Windows环境下正常运行
- **依赖管理**：明确列出所有依赖，提供安装说明
- **错误处理**：脚本必须有完善的错误处理和用户提示
- **性能要求**：笔记验证时间不超过30秒，错题处理不超过2分钟

### 兼容性要求
- **Markdown渲染**：确保在GitHub、Typora、VS Code等环境正确显示
- **数学公式**：支持MathJax和KaTeX渲染引擎
- **思维导图**：确保Mermaid在各种环境下正确渲染
- **移动设备**：内容在手机和平板上可读性良好

## 治理规则 (Governance)

### 宪法地位
- **最高优先级**：本宪法优先于所有其他规范和流程
- **修改程序**：宪法修改需要详细说明、充分讨论、正式批准
- **执行监督**：所有PR和审查必须验证宪法合规性
- **争议解决**：如有冲突，以宪法条款为准

### 质量保证
- **自动化验证**：所有提交必须通过自动化质量检查
- **人工审核**：重要内容变更必须有专人审核
- **持续改进**：基于使用数据和反馈持续优化系统
- **培训要求**：所有参与者必须熟悉本宪法和相关规范

### 责任分工
- **内容创建者**：负责内容准确性和格式规范性
- **审核者**：负责质量把关和合规性检查
- **系统维护者**：负责工具更新和流程优化
- **用户**：负责及时反馈问题和改进建议
